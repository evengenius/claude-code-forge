# Рецензии Claude Code Forge

Внешние рецензии v1.0.0 от ChatGPT и Gemini, с комментариями автора.

---

## Рецензия ChatGPT

### Выявленные слабые места

1. **Привязка к Claude (vendor lock-in)** — методология завязана на Claude Code.
   - *Комментарий*: формально верно, но мимо цели. CLAUDE.md, хуки PostToolUse/PreToolUse/Stop,
     `/compact`, субагенты, path-scoped rules — это API Claude Code. «Абстрактный LLM adapter»
     превратит методологию в набор общих советов. Принципы (TDD, memory bank, ADR) универсальны.
     Реализация — нет, и это нормально.

2. **Отсутствие строгих гарантий безопасности** — deny-list это соглашение, не защита.
   - *Комментарий*: **самый сильный пункт**. permissions.deny блокирует прямой Read(.env),
     но Claude может получить содержимое косвенно: через `cat .env`, `grep PASSWORD .`,
     через stderr приложения. DevContainer — реальное решение. **Принято в roadmap v2.1.0.**

3. **Нет формального feedback loop** — система не проверяет качество автоматически.
   - *Комментарий*: Stop-хук запускает тесты, `/project:review` проводит code review.
     Полный CI — задача проекта, не bootstrap-скрипта.

4. **Перегрузка контекста** — много правил и файлов.
   - *Комментарий*: уже решено: CLAUDE.md — 40-60 строк, rules загружаются по path-scope,
     memory bank читается по команде, не подгружается целиком.

5. **Зависимость от дисциплины пользователя** — методология требует строгого соблюдения.
   - *Комментарий*: **справедливо**. Handoff, обновление memory bank — всё ручное.
     **Принято в roadmap v2.2.0** (git-хук напоминание).

6. **Отсутствие метрик эффективности** — нет измеримых показателей.
   - *Комментарий*: задача отдельного инструмента, не bootstrap-скрипта.

7. **Монолитный bootstrap-скрипт** — 1200 строк bash, сложно расширять.
   - *Комментарий*: **справедливо**. **Принято в roadmap v2.0.0** (модуляризация).

### Итог по ChatGPT
Качественный структурный анализ с enterprise-креном. Анализирует Forge как будто это
«управляемый engineering framework», а не утилита для соло-разработчика.

## Рецензия Gemini (первая попытка)

Gemini ошибочно принял Forge за Python-утилиту сборки файлов проекта (контекст-коллектор
типа repomix/code2prompt). Оценка 7/10 и все выводы (счётчик токенов, инкрементальные
обновления) относились к другому проекту. **Рецензия нерелевантна.**

## Рецензия Gemini (вторая попытка, после коррекции)

После указания на ошибку Gemini дал предметный разбор трёх принятых пунктов:

1. **Безопасность: от «джентльменских соглашений» к изоляции**
   - Предложил генерацию Dockerfile / devcontainer.json
   - Идея проверки «запущен ли я в контейнере?» при старте
   - *Комментарий*: точно. Уточнение: permissions.deny — не мягкая инструкция в промпте,
     а проверка на уровне tool use. Проблема в косвенном доступе, не в игнорировании правил.

2. **Проблема «Пятничного Хэндоффа»**
   - Git-хук (post-commit) проверяющий обновление activeContext.md
   - Алиас на exit, спрашивающий «А ты сделал handoff?»
   - *Комментарий*: элегантное минимальное решение.

3. **Модульность: смерть монолита**
   - Структура: src/core.sh + src/templates/ + src/stacks/
   - Контрибьюторы добавляют стеки не ломая core
   - *Комментарий*: правильная декомпозиция, принята.

### Финальный вопрос Gemini
«Стоит ли Memory Bank выносить в переключаемый пресет или оставить жёстким ядром?»

**Ответ: ядро.** Memory Bank решает корневую проблему (потеря контекста между сессиями).
Без него — просто генератор папок. Переключаемыми пресетами должны быть стеки, хуки,
команды — всё про «как пишем код». Memory Bank — про «как сохраняем знание».
Для Minimal-уровня можно свернуть до двух файлов (activeContext + progress).
